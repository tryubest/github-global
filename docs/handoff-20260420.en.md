### 5. Pitfalls encountered and compromises

**Pitfall 1: Tool call interruption loop**

- Phenomenon: Multiple `Read was interrupted by the user after XXXms` errors appeared in the session, interrupting the workflow
- Root cause inference: Concurrent 3+ Read calls reading different fragments of the same file may have triggered IDE suspension/timeout
- Remediation: Later **no longer do fragmented parallel reading** for already familiar files, instead do one Read of the entire file + locate in memory, or directly use StrReplace for precise changes
- **Migratable experience**: Cursor Agent cannot self-resume after being interrupted; must be triggered by user sending a new message; therefore better to spend one extra tool call for "block reading" rather than betting on concurrency saving time

**Pitfall 2: Misunderstanding of Persona C's permissions**

- Initial PRD described Persona C as "non-paying beneficiary", but author's intuition was "Persona C can also actively submit translations"
- Actually not feasible under GitHub App permission model - only repository owners can grant Installation
- Solution: Explicitly write **scope description + why** at the end of PRD 2.3 and 3.1, and add Fork-first (F-24) as Persona C's legitimate participation channel in P2
- **Migratable experience**: When doing third-party platform SaaS, first ask "who → who in authorization direction", then design roles, otherwise PRD will be rewritten halfway through

**Compromise Record**

- **Data minimization** vs **debugging convenience**: MVP stage allows keeping original/translated full text (for troubleshooting AI translation errors), will be removed before GA. Declared in 6.1 security table
- **Queue solution**: MVP uses database table to simulate queue, no Redis/Upstash. Saves cost but has low throughput ceiling, needs replacement when volume increases
- **Cost model**: Learning period uses all `:free` models, delivery experience may not be as good as paid models. But fits project positioning (learning + startup), acceptable
