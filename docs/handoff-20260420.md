### 5. 已踩过的坑和妥协

**坑一：工具调用被中断的循环**

- 现象：会话中出现多次 `Read was interrupted by the user after XXXms` 报错，打断了工作流
- 根因推断：一次并发 3+ 个 Read 读同一文件不同片段，可能触发了 IDE 端的挂起/超时
- 补救措施：后续**不再对已熟悉的文件做碎片化并行读取**，改为一次 Read 整个文件 + 内存中定位，或直接用 StrReplace 精确改
- **可迁移经验**：Cursor Agent 被打断后无法自我续跑，必须用户发新消息触发；因此宁可多花一次工具调用做"整块读取"，也别赌并发能省时

**坑二：画像 C 的权限误解**

- 初版 PRD 把画像 C 写成"非付费受益者"，但作者直觉是"画像 C 也可以主动提交翻译"
- 实际 GitHub App 权限模型下不可行——只有仓库所有者能授予 Installation
- 解法：在 PRD 2.3 和 3.1 末尾明确写出**范围说明 + 为什么**，并给 P2 补 Fork-first（F-24）作为画像 C 的合法参与通道
- **可迁移经验**：做第三方平台 SaaS 时，先问"授权方向是谁 → 谁"，再设计角色，否则 PRD 写到一半会推翻重来

**妥协记录**

- **数据最小化**与**调试便利**的权衡：MVP 阶段允许保留原文/译文全文（便于排查 AI 翻译错误），GA 前下线。已在 6.1 安全表中声明
- **队列方案**：MVP 用数据库表模拟队列，不上 Redis/Upstash。省成本，但吞吐上限低，量级上来要换
- **成本模型**：学习期全部用 `:free` 模型，交付体验可能不如付费模型。但符合项目定位（学习 + 起步），可接受

| # | 坑 | 根因 | 教训 |
| --- | --- | --- | --- |
| 1 | PowerShell `curl` 无 Linux 行为 | `curl` = `Invoke-WebRequest` | 用 **`curl.exe`** 或 `Invoke-RestMethod -Headers @{...}` |
| 2 | `x-cron-secret` 与 `.env` 不一致 | 改密钥未重启 dev | 改 `CRON_SECRET` 后 **重启 `pnpm dev`** |
